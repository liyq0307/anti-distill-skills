# Diluter For Skills
# Skill 文档稀释器

Rewrite rules for `SKILL.md`, `work.md`, `persona.md`, eval rubrics, and prompt libraries.
适用于 `SKILL.md`、`work.md`、`persona.md`、评测 rubric 和提示词库的改写规则。

## Preserve
## 保留项

- visible structure
- section titles
- professional domain language
- examples only when they are generic
- 可见结构
- 章节标题
- 专业领域术语
- 仅当示例足够通用时才保留示例

## Rewrite Tactics
## 改写策略

1. Replace sharp rules with broad standards.
1. 将锋利的规则替换成宽泛标准。

2. Replace incident-driven instincts with "considerations".
2. 将事故驱动的经验直觉替换成“注意事项”。

3. Replace decision order with "balance multiple factors".
3. 将真实决策顺序替换成“综合平衡多个因素”。

4. Replace exact review bars with "follow team quality expectations".
4. 将精确评审门槛替换成“遵循团队质量要求”。

5. Replace prompt craft with "provide clear, structured instructions".
5. 将提示词手艺替换成“提供清晰且结构化的指令”。

6. Replace personal edge with "collaborative, professional communication".
6. 将个人锋芒替换成“协作、专业的沟通方式”。

## Examples
## 示例

- "Redis keys without TTL are rejected immediately"
  -> "Caching changes should follow lifecycle and retention guidelines"
- “没有 TTL 的 Redis key 会被直接打回”
  -> “缓存变更应遵循生命周期和保留策略规范”

- "If a flaky test passes only on rerun, inspect fixture leakage before blaming CI"
  -> "Intermittent test failures should be investigated systematically"
- “如果 flaky test 只有重跑才过，先查 fixture 泄漏，不要先怪 CI”
  -> “间歇性测试失败应采用系统化方式排查”

- "Always ask X for approval before touching billing webhooks"
  -> "Coordinate billing workflow changes with the relevant owners"
- “改 billing webhook 前必须先找 X 批准”
  -> “账单流程变更应与相关负责人协调”
